## 2. MVVM + Repository Pattern

### Models

- Immutable data classes generated by the `freezed` package.
- Each model has a `fromJson` / `toJson` factory for Firestore serialization.
- Models live in `features/<feature>/models/`.
- Models contain NO business logic, NO Firestore calls.

### Repositories

- Each feature has an **abstract repository** (`*_repository.dart`) and a **Firebase implementation** (`*_repository_impl.dart`), both in the `models/` folder.
- The abstract class defines the contract (method signatures only). The implementation contains all Firebase/Firestore calls.
- This separation allows swapping the backend or mocking for tests without touching ViewModels or Views.
- Repositories are provided via Riverpod `Provider` so ViewModels access them through `ref.read()`.

#### Example

```dart
// features/auth/models/auth_repository.dart
abstract class AuthRepository {
  Future<void> signUp(String email, String password);
  Future<void> signIn(String email, String password);
  Future<void> signOut();
  Future<void> resetPassword(String email);
  Stream<User?> authStateChanges();
}

// features/auth/models/auth_repository_impl.dart
class AuthRepositoryImpl implements AuthRepository {
  final FirebaseAuth _auth;
  final FirebaseFirestore _firestore;
  // ... Firebase implementation
}

// Provider
final authRepositoryProvider = Provider<AuthRepository>(
  (ref) => AuthRepositoryImpl(
    FirebaseAuth.instance,
    FirebaseFirestore.instance,
  ),
);
```

### ViewModels

- Riverpod `AsyncNotifier` or `Notifier` classes.
- Each ViewModel manages state for one screen or one logical unit of work.
- ViewModels call the **repository** (via its abstract interface) to fetch and mutate data. ViewModels never import `cloud_firestore` directly.
- ViewModels expose state as `AsyncValue<T>` for loading/error/data handling.
- ViewModels live in `features/<feature>/viewmodels/`.

### Views

- Flutter widgets that consume ViewModel state via `ref.watch()`.
- Views call ViewModel methods via `ref.read()` in response to user actions.
- Views handle ONLY presentation: layout, styling, animations.
- Views NEVER import `cloud_firestore`, repositories, or contain business logic.
- Views live in `features/<feature>/views/`.

### Data Flow

```
View → (user action) → ViewModel → Repository (abstract) → RepositoryImpl (Firebase)
View ← (state update) ← ViewModel ← Repository ← Firestore
```

## 3. Riverpod Usage

### Provider Types

| Use Case                     | Provider Type           |
| ---------------------------- | ----------------------- |
| Screen state (async data)    | `AsyncNotifierProvider` |
| Screen state (sync data)     | `NotifierProvider`      |
| Firebase Auth stream         | `StreamProvider`        |
| Current user document stream | `StreamProvider`        |
| One-off reads                | `FutureProvider`        |
| Simple computed values       | `Provider`              |

### Conventions

- All providers are declared at the top of the ViewModel file.
- Provider names use camelCase: `matchFeedViewModelProvider`.
- ViewModels extend `AsyncNotifier<T>` or `Notifier<T>`.
- The `build()` method contains the initial data fetch.
- Mutations are instance methods on the ViewModel class.

  }
}

## 4. go_router Configuration

### Route Structure

```
/                         → Welcome screen (unauthenticated)
/sign-in                  → Sign In screen
/sign-up                  → Sign Up screen
/forgot-password          → Forgot Password screen
/setup                    → Profile Setup Wizard
/home                     → Match Feed (inside ShellRoute)
/explore                  → Explore / Search (inside ShellRoute)
/profile/:userId          → User Profile (inside ShellRoute)
/profile/edit             → Edit Own Profile
/barter-requests          → Barter Requests List (inside ShellRoute)
/barter-room/:barterId    → Barter Room
/review/:barterId         → Review Form
```

### ShellRoute

The `ShellRoute` wraps the main app screens with a persistent navigation shell:

- **Mobile (< 700px)**: `BottomNavigationBar` with tabs: Home, Explore, Requests, Profile.
- **Web (≥ 700px)**: `NavigationRail` or sidebar with the same tabs.

The shell is defined in `shell_screen.dart` and uses `ResponsiveLayout` to decide which navigation type to render.

### Auth Guard

A `redirect` function on the `GoRouter` checks:

1. If user is not authenticated → redirect to `/`.
2. If user is authenticated but `isProfileComplete == false` → redirect to `/setup`.
3. Otherwise → allow navigation.

### Navigation Rules

- Always use `context.go()` or `context.push()`.
- Never use `Navigator.push()` or `Navigator.of(context)`.

## 5. Responsive Layout

### Breakpoint

| Width   | Layout      |
| ------- | ----------- |
| < 700px | Mobile      |
| ≥ 700px | Web/Desktop |

### Implementation (`responsive_layout.dart`)

```dart
class ResponsiveLayout extends StatelessWidget {
  final Widget mobileLayout;
  final Widget desktopLayout;

  const ResponsiveLayout({
    required this.mobileLayout,
    required this.desktopLayout,
  });

  @override
  Widget build(BuildContext context) {
    return LayoutBuilder(
      builder: (context, constraints) {
        if (constraints.maxWidth >= 700) {
          return desktopLayout;
        }
        return mobileLayout;
      },
    );
  }
}
```

### Usage

- `ShellScreen` uses `ResponsiveLayout` to switch between `BottomNavigationBar` (mobile) and `NavigationRail` (desktop).
- Individual screens can also use `ResponsiveLayout` to adjust their content layout (e.g., single-column on mobile, two-column on desktop).

## 6. Firebase Services

### Firebase Auth

- Email/password sign up and sign in.
- Password reset via `sendPasswordResetEmail()`.
- Auth state is observed via `authStateChanges()` stream → `StreamProvider`.
- On sign up, a Firestore user document is created with `isProfileComplete: false`.

### Cloud Firestore

- All data persistence.
- Collections: `users`, `barters`, `reviews`, `matches`.
- All reads/writes go through **repository implementations**, called by ViewModels. Views and ViewModels never import `cloud_firestore`.
- Gender filter is applied at the query level (`.where('gender', isEqualTo: ...)`).
- All timestamps use `FieldValue.serverTimestamp()`.

### Firebase Cloud Messaging (FCM)

- Push notifications for barter requests, messages, session reminders.
- FCM token is stored on the user document.
- Notifications are sent by Cloud Functions (not from the client app).

### Firebase Cloud Functions (Node.js / JavaScript)

- `computeMatchEmbeddings`: Triggered when a user's `canTeach` or `wantsToLearn` changes. Calls OpenAI API to compute text embeddings for the user's skills. Stores the embedding in `users/{userId}.matchEmbedding`.
- `computeMatchScores`: Triggered after embeddings are updated. Computes cosine similarity between the updated user and all same-gender users. Writes results to `matches/{userId}/suggestions/{matchedUserId}`.
- `sendNotification`: Triggered on Firestore writes (new barter request, barter status change, new message). Sends FCM push notification to the relevant user.
- `sessionReminder`: Scheduled function that checks for barters with sessions starting in the next hour and sends reminder notifications.

## 7. AI Matching — Cloud Function

### How It Works

1. **Trigger**: A user updates their `canTeach` or `wantsToLearn` array in Firestore.
2. **Embedding**: The Cloud Function concatenates the user's `canTeach` and `wantsToLearn` into a single text string and calls the OpenAI `text-embedding-3-small` model to generate a vector embedding.
3. **Storage**: The embedding vector is stored in `users/{userId}.matchEmbedding`.
4. **Scoring**: A second function (or the same function) queries all same-gender users who also have embeddings. For each pair, it computes cosine similarity between the two embedding vectors.
5. **Result**: Match scores (0–100) are written to `matches/{userId}/suggestions/{matchedUserId}` with a `score` field.
6. **App reads**: The Flutter app reads from `matches/{currentUserId}/suggestions` ordered by `score` descending to populate the match feed.

### Cosine Similarity

```javascript
function cosineSimilarity(vecA, vecB) {
  let dotProduct = 0;
  let normA = 0;
  let normB = 0;
  for (let i = 0; i < vecA.length; i++) {
    dotProduct += vecA[i] * vecB[i];
    normA += vecA[i] * vecA[i];
    normB += vecB[i] * vecB[i];
  }
  return dotProduct / (Math.sqrt(normA) * Math.sqrt(normB));
}
```

### Important Notes

- The app NEVER computes match scores. It only reads precomputed scores.
- Embeddings are NEVER sent to or stored on the client.
- Only same-gender matches are computed (gender filter is applied in the Cloud Function too).

## 9. Theming — Light & Dark Mode

### Architecture

- `app_colors.dart` defines two static color palettes: `AppColors.light` and `AppColors.dark`.
- `app_theme.dart` builds `ThemeData` for both modes using the color palettes and `AppTextStyles`.
- `app.dart` passes both `theme` and `darkTheme` to `MaterialApp.router`, using `themeMode: ThemeMode.system` to follow the OS setting.
- All widgets use `Theme.of(context).colorScheme` or the custom `AppColors` extension on `BuildContext` — never hardcoded values.

### Example

```dart
// core/theme/app_colors.dart
class AppColors {
  // Light palette
  static const light = AppColorPalette(
    primary: Color(0xFF...),
    background: Color(0xFF...),
    surface: Color(0xFF...),
    // ...
  );

  // Dark palette
  static const dark = AppColorPalette(
    primary: Color(0xFF...),
    background: Color(0xFF121212),
    surface: Color(0xFF1E1E1E),
    // ...
  );
}

// core/theme/app_theme.dart
class AppTheme {
  static ThemeData light() => ThemeData(
    colorScheme: ColorScheme.fromSeed(
      seedColor: AppColors.light.primary,
      brightness: Brightness.light,
    ),
    // ...
  );

  static ThemeData dark() => ThemeData(
    colorScheme: ColorScheme.fromSeed(
      seedColor: AppColors.dark.primary,
      brightness: Brightness.dark,
    ),
    // ...
  );
}
```

## 11. Figma Workflow

Before starting any UI task, the user will provide the Figma URL for the relevant screens. The implementation must match the Figma design exactly — colors, spacing, typography, layout, and component styles. If the Figma design conflicts with the plan (e.g., uses a color not in `AppColors`), add the color to `AppColors` to match the design. The Figma is the source of truth for all visual decisions.
